# README - Proyecto: Login & Chat - Coding Dojo Frontend (Sprint 1)

## 1. Visión General del Proyecto

Este proyecto consiste en el desarrollo de una interfaz de Login y Chat para aplicaciones. El objetivo principal de la fase de Login es que el **usuario pueda ver un formulario para ingresar sus credenciales de autenticación**.

El trabajo de planificación inicial incluyó un Kick-off, la presentación del tablero y el Setup del repositorio y la estructura base.

---

## 2. Sprint 1: Maquetado del Formulario HTML

| Aspecto | Detalle |
| :--- | :--- |
| **Duración** | 0:30 - 1:00 (30 minutos) |
| **Objetivo** | **Maquetar el formulario HTML**. |
| **Equipo Responsable** | TEAM |
| **Resultado de la Review** | Presentación del resultado de Sprint, específicamente el archivo `Login.html`. |

### Historia de Usuario (HU1)

El foco del Sprint 1 fue la Historia de Usuario 1 (HU1): Formulario de login.

**Como** usuario de mis aplicaciones, **Quiero** ver un formulario de Login, **Para** ingresar mis credenciales de autenticación.

### Criterios de Aceptación Implementados (HU1)

El formulario HTML construido satisface los siguientes criterios:
*   El formulario debe tener campos para usuario y contraseña.
*   Debe incluir un botón Login.
*   Los campos deben estar etiquetados claramente.

### Requerimientos / Configuración Inicial

Para dar inicio al desarrollo, se cumplieron los siguientes requerimientos y pasos previos:
*   Creación del repositorio GitHub por el Líder Técnico (LT).
*   Selección de la **estrategia Gitflow**.
*   Establecimiento de la estructura base del repositorio.
*   Finalizado el sprint, se realizó la **actualización del Read.me** y del código en la rama del repositorio de origen.

## 2. Sprint 2: Implementación de Estilos CSS y Diseño Responsive

El objetivo del Sprint 2 fue enfocar el trabajo del equipo (TEAM) en la aplicación de **Estilos CSS**.

El trabajo se centró en la Historia de Diseño 2 (HD2): Diseño responsive.

| Rol | Tarea |
| :--- | :--- |
| **Como** usuario de login | **Quiero** que el diseño sea atractivo y responsivo |
| **Para** | tener una mejor experiencia de usuario. |

### Criterios de Aceptación Implementados (HD2)

El diseño del formulario de Login debe cumplir con los siguientes puntos para mejorar la experiencia de usuario:

Para poder iniciar este sprint, fue un requerimiento fundamental contar con el **Témplate HTML ya Construido** durante el Sprint 1.

La presentación del resultado del Sprint consiste en la entrega del archivo **`Login.css`**.

Una vez finalizado el Sprint 2 y habiendo establecido el diseño responsive, el siguiente paso en la planificación es el **Sprint 3: Validación JS**.

## 3. Sprint 3: Validación con JavaScript (JS)

El Sprint 3 se enfocó en la implementación de la **Validación JS** por parte del equipo (TEAM).

Este sprint tuvo una duración programada de 45 minutos, desarrollándose entre las 2:15 y las 3:00.

### Historia de Usuario (HU3): Validación Campos Vacíos

El objetivo de este sprint fue abordar la Historia de Usuario 3 (HU3): Validación campos vacíos.

| Rol | Tarea |
| :--- | :--- |
| **Como** usuario del Login | **Quiero** recibir un mensaje si dejo campos vacíos |
| **Para** | evitar que se ingresen los datos vacíos. |

### Criterios de Aceptación Implementados (HU3)

Para asegurar una mejor experiencia de usuario y la integridad de los datos, la lógica implementada en este sprint cumple con:

1.  Si el campo de usuario o el de contraseña están vacíos, se debe mostrar el mensaje: **"Por favor complete todos los campos"**.
2.  El mensaje de error generado debe ser **visible y en color rojo**.
3.  La validación de *inputs* es un requisito clave para la puntuación general del proyecto (5 puntos).

### Requerimientos y Entregables

La implementación de la validación con JavaScript dependió de los entregables de los sprints anteriores:

*   **Requerimientos:** Se requirió el **Témplate HTML ya Construido (Sprint 1)** y la **Creación del archivo CSS (Sprint 2)**.
*   **Entregable:** La creación del **archivo JS** fue un requerimiento fundamental para ejecutar la lógica de validación de campos.

## 4. Sprint 4: Autenticación Simulada (JS)

El Sprint 4 se enfocó en la implementación de la **Autenticación** y la lógica de simulación de acceso por parte del equipo (TEAM).

Este sprint tuvo una duración de 30 minutos, desarrollándose entre las 3:00 y las 3:30.

### Historia de Usuario (HU4): Autenticación Simulada

El objetivo principal de este sprint fue abordar la Historia de Usuario 4 (HU4): **Autenticación simulada**.

### Criterios de Aceptación Implementados (HU4)

La lógica implementada en JavaScript simula un proceso de validación de credenciales:

1.  Se debe **Validar el Usuario según una lista** predefinida de credenciales.
2.  La lista de credenciales utilizada para la validación es: **{Admin/1234, Ginett/2345, Oscar/3456, Cristian/4567, Luis/5678, Felipe/6789}**.
3.  El sistema debe mostrar un **Mensaje de éxito o de error** tras intentar la autenticación.
4.  La correcta **Simulación de interacción (mock)** tiene un valor de **4 puntos** en la puntuación total del proyecto.

### Requerimientos y Entregables

La funcionalidad de autenticación dependió de las entregas de los sprints anteriores:

*   **Requerimientos:** El desarrollo se construyó sobre el Témplate HTML ya Construido (Sprint 1), la Creación del archivo CSS (Sprint 2) y la lógica de Validación JS (Sprint 3),.
*   **Entregable:** La implementación de esta lógica funcional resultó en la actualización del archivo JavaScript.
